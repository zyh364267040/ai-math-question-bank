# SQLite 数据库设计

## 一、设计目标

第一版数据库服务于天津高中数学题目的电子化、分类、审核、选题和使用记录。结构优先保存可追溯的原始事实，将多值标签和事件记录拆分到关联表，并用稳定代码与数字主键兼顾人工使用和数据库关联效率。

本版仅使用 Python 标准库 `sqlite3`。SQLite 连接在建表和日常使用时都必须开启 `foreign_keys`。时间字段保存 ISO 8601 文本；数据库默认时间使用 SQLite `CURRENT_TIMESTAMP`，业务代码写入时应明确时区。

## 二、表职责

### 题目与内容表

- `questions` 是题目主表，保存唯一业务编号、题干、必填答案、可选解析、来源、单值分类、各审核项最新状态、整体可用状态、内容哈希和重复组归属。
- `question_options` 保存选择题的选项代码、内容和顺序。
- `subquestions` 保存解答题小问的题干、答案、可选解析、分值和顺序。
- `question_formulas` 保存公式级检索与审核需要的 LaTeX、出现位置和顺序，题干中仍保留原位公式。
- `question_figures` 保存题目或小问配图的相对路径、用途、顺序、替代文本、来源类型和图像哈希。

### 知识点与标签表

- `knowledge_points` 保存数字内部主键、人可读稳定代码、名称、层级、父节点、同级排序、启用状态和体系版本。
- `questions.primary_knowledge_point_id` 保存唯一主知识点。
- `question_related_knowledge_points` 保存题目与关联知识点的多对多关系。
- `tag_definitions` 统一管理解题任务、解题方法、易错点和适用场景四类标签，每类使用稳定代码。
- `question_tags` 保存题目与标签的多对多关系，并允许为易错点等关联增加简短说明。

### 字典、质量和历史表

- `regions`、`exam_types`、`question_types`、`review_statuses`、`usability_statuses`、`difficulty_levels` 和 `version_statuses` 提供受控值。
- `question_reviews` 保留每次审核的项目、前后状态、审核人文本、时间和备注；题目主表只保存最新状态。
- `question_usage_records` 保存已经发生的课堂、练习等使用事件。是否使用、使用次数和最近使用时间由该表查询聚合，不在主表维护可能失配的缓存。
- `duplicate_groups` 表示人工确认或待确认的重复题组，同组题目仍可分别保留来源事实。
- `question_versions` 按题目保存版本号、状态、上一版本和 JSON 快照，用于内容实质修改后的追溯。

## 三、关键关系和删除策略

题目通过内部整数 ID 关联子表，通过 `question_code` 供人工检索和跨系统引用。题目删除时，选项、小问、公式、配图关联、关联知识点、标签关联、审核历史、使用记录和版本快照级联删除。

知识点、标签和受控字典被题目引用时使用 `RESTRICT`，防止物理删除破坏历史数据。不再使用的知识点或标签应设为停用。删除重复组只清空题目的组归属，不删除题目。

数据库结构保证知识点引用存在。“主知识点必须是三级”、“关联知识点不得与主知识点重复”等需要跨表查询的规则，由后续入库服务在同一事务中校验。

## 四、约束

- 所有主表和事件表都有主键；多对多关联使用联合主键防止重复。
- `questions.question_code` 全库唯一，知识点稳定代码、重复组代码和字典代码也有唯一约束。
- 题干和答案不允许空字符串，详细解析可空。解析存在时，解析审核状态不能是“不适用”；解析不存在时必须是“不适用”。
- 题型、难度、各审核状态、整体可用状态和版本状态通过外键限定为基础字典中的值。
- 难度层级和知识点层级有数值范围检查，分值不能为负，布尔状态只能为 0 或 1，顺序和版本号必须为正整数。
- 配图和原始文件保存相对路径，结构层拒绝以 `/` 开头的 POSIX 绝对路径；跨平台完整路径检查由入库服务补充。
- 题目设为 `usable` 时，答案和标签审核必须通过，OCR、公式和配图项必须通过或不适用，已有解析必须通过审核。
- 审核历史发生时，审核人为非空自由文本，本版不依赖用户表。

SQLite 不会对每个新连接永久保存 `foreign_keys` 开关。因此 `schema.sql` 显式设置该开关，`initialize_database` 也会在连接建立后立即开启。后续其他数据库连接入口也必须做同样设置。

## 五、索引

- 题目内容哈希索引用于精确重复候选检索，但不设唯一，因为同题在不同真实试卷的出现事实需要分别保留。
- 地区、年份、考试类型、试卷名和原题号组合索引支持来源追溯与考频统计。
- 主知识点、关联知识点和标签反向索引支持分类选题。
- 题型与难度组合索引、可用状态索引和重复组索引支持常用筛选。
- 审核和使用记录按“题目 + 时间”建立索引，支持历史查询、使用次数和最近使用时间聚合。

## 六、基础字典

初始化脚本使用 `INSERT OR IGNORE` 写入稳定代码，可安全重复执行。首版包含天津地区，高考、一模、二模、月考、期中、期末、教材、教辅、教师自编、学生错题和其他考试类型，单选、多选、填空和解答四种题型，四种审核状态，五种整体可用状态，五级难度，以及字段与标签标准中的第一版解题任务、方法、易错点和适用场景。

知识点种子位于 `data/samples/knowledge_points_v1.json`，属于可公开复核的领域样例数据，不包含试题、学生数据或版权材料。该文件由 `src/knowledge/parse_knowledge_system.py` 读取 `docs/domain/高中数学高考知识点体系.md` 后生成，不在 Python 中复制知识体系正文。需要更新种子时，应先修改 Markdown 来源，再运行解析器重新生成 JSON，并审核差异。

### 知识点稳定代码

一级模块直接沿用 Markdown 的两位编号 `01` 至 `11`。二级专题直接沿用 Markdown 编号，例如 `01.01` 和 `08.06`。三级知识点按其在所属二级专题中的出现顺序，从 `01` 开始生成，例如 `01.01.01`。显示名称与稳定代码分离；只修改名称而不改变节点位置时，代码保持不变。

`sort_order` 表示节点在同一父级下的顺序，从 1 开始。一级模块按模块编号排序，二级专题按专题编号的末段排序，三级知识点按项目符号出现顺序排序。

### 解析和校验边界

解析器只读取 `## 01` 至 `## 11` 模块中的二级标题和项目符号。模块 11 后的“题库标签使用规则”等章节不会进入种子。

生成前必须通过结构校验：一级模块完整包含 01 至 11 且编号不重复；二级专题编号必须以当前一级模块编号开头且全局不重复；每个二级专题至少包含一个三级知识点；同一专题内三级名称不得重复；所有非一级节点的父级代码必须存在；所有节点代码必须唯一。任何校验失败都应中止生成并报告具体编号或名称，不允许静默跳过。

### 导入和更新规则

初始化器按一级、二级、三级的父级顺序导入。数据库以 `knowledge_points.code` 为冲突键：新代码插入新记录，已有代码更新名称、父级、层级、排序和体系版本，因此重复初始化不会创建重复记录，名称修订也不会产生新知识点。内部整数 ID 只用于数据库关联，不作为跨库种子标识。

种子中移除的代码目前不会被初始化器删除，也不会自动停用，以免破坏历史题目关联。未来确需合并、废弃或改码时，应新增显式版本迁移，记录旧代码到新代码的映射，先迁移引用并经人工复核，再决定是否设置 `is_active = 0`；不得依靠普通初始化执行物理删除。

## 七、初始化方法

在项目根目录运行 `python3 src/database/initialize.py`，数据库默认创建为 `data/private/question-bank.db`。初始化会同时幂等写入基础字典和已审核的知识点 JSON 种子。`data/private/*` 和常见 SQLite 扩展名已由 `.gitignore` 排除，运行数据库不应提交 Git，目录中只保留 `.gitkeep`。

如需指定路径，将路径作为唯一位置参数，例如 `python3 src/database/initialize.py /tmp/question-bank-test.db`。脚本会创建父目录，重复执行只补齐结构和缺失字典，不重复写入数据。测试始终在系统临时目录创建数据库，不会污染 `data/private`。

初始化器同时创建 PDF 入口所需的 `source_papers` 和 `import_jobs`。前者以完整 SHA-256 唯一标识一份原始文件，只保存相对于私有数据根目录的安全路径；后者记录每次处理请求和页码范围，并用 CHECK 约束 `pending`、`processing`、`needs_review`、`completed`、`failed` 五种状态。详细操作边界和失败恢复规则见《PDF导入与审核流程》。

## 八、本次采用的五项默认决定

1. 知识点同时保存 `INTEGER` 内部主键和人可读稳定 `code`。表关联使用数字 ID，展示、导入和跨系统交流使用稳定代码。
2. 第一版答案必填，详细解析可后补。已有解析必须进入审核状态，且题目正式可用时解析必须审核通过。
3. 第一版审核人使用 `question_reviews.reviewer` 自由文本，不建立用户表。
4. 考频按每张真实试卷中的出现事实计数，相同内容可以作为不同题目记录保留来源。选题去重时使用 `duplicate_groups`，不用内容哈希唯一约束丢失出现事实。
5. 第一批没有学生作答数据，难度先依据教师判断、题位和分值，可在 `difficulty_basis` 保留依据。后续增加学生正确率和用时数据后再校准难度，不将首版判断视为永久事实。
